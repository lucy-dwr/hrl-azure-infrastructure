# Naming Conventions

This document covers naming patterns for Azure resources, Terraform constructs, and related files in this repository. Follow these patterns consistently so names are predictable and self-describing.

## General principles

- Use lowercase letters, numbers, and hyphens wherever Azure allows.
- Include `hrl`, the workload, the environment, and the region in every resource group name.
- For resources where Azure prohibits hyphens (storage accounts), concatenate tokens without separators.
- For resources where a name must be globally unique (storage accounts), append a short random suffix.
- Never include data type categories (`spatial`, `raster`, `tabular`) in infrastructure names. Data type organization belongs in storage paths, catalogs, and schemas.

## Environments

Valid environment values are:

| Value   | Meaning                    |
|---------|----------------------------|
| `dev`   | Developer / local testing  |
| `test`  | Automated test environment |
| `stage` | Pre-production staging     |
| `prod`  | Production                 |

## Region codes

| Azure region | Code used in names |
|--------------|--------------------|
| `westus3`    | `wus3`             |
| `westus2`    | `wus2`             |

`westus3` is the default region. Some services (Azure Static Web Apps) are not available in `westus3` and use `westus2` instead; the resource name still omits the region segment in those cases.

## Workloads

| Workload     | Description                                           |
|--------------|-------------------------------------------------------|
| `tfstate`    | Terraform remote state storage (bootstrap only)       |
| `core`       | Shared foundation resources (Key Vault, Log Analytics, managed identities) |
| `data`       | Durable data storage (ADLS Gen2, storage containers)  |
| `pipelines`  | Ingestion and processing resources                    |
| `apps`       | Public and partner-facing applications                |
| `ai`         | LLM inference resources (Microsoft Foundry / Azure AI Foundry accounts, projects, model deployments) |

## Resource groups

Pattern:

```
rg-<program>-<workload>-<environment>-<region>
```

Examples:

```
rg-hrl-tfstate-prod-wus3
rg-hrl-core-prod-wus3
rg-hrl-data-prod-wus3
rg-hrl-pipelines-prod-wus3
rg-hrl-apps-prod-wus3
rg-hrl-ai-prod-wus3
```

## Azure resource names

### Storage accounts

Azure storage account names must be globally unique, 3–24 characters, lowercase letters and numbers only (no hyphens). Append a six-character random alphanumeric suffix to guarantee uniqueness.

Pattern:

```
st<program><workload><environment><random>
```

Example:

```
sthrltfstateprodr4x8kz
```

The random suffix is generated by `random_string` in Terraform and is stable once the resource is created.

### Storage containers / blob containers

Use a short, lowercase, hyphen-separated name that describes the purpose.

Examples:

```
tfstate
raw-submissions
public-exports
```

### Static Web Apps

The default region for Static Web Apps is `westus2` (`wus2`). Include the region segment using the `wus2` code.

Pattern:

```
stapp-<program>-<app-slug>-<environment>-<region>
```

Example:

```
stapp-hrl-restoration-map-prod-wus2
```

### Key Vault

Pattern:

```
kv-<program>-<workload>-<environment>-<region>
```

Example:

```
kv-hrl-core-prod-wus3
```

Key Vault names must be globally unique, 3–24 characters.

### Microsoft Foundry (Azure AI Foundry) account

Pattern:

```
cog-<program>-<workload>-<environment>-<random>
```

Example:

```
cog-hrl-ai-prod-a7f2mq
```

The account name must be globally unique (it doubles as the custom subdomain), so it appends the same short random suffix pattern used for storage accounts. The account is created as a Cognitive Services account with `kind = "AIServices"` and `project_management_enabled = true`, which is what makes it a Foundry account rather than a single-service Cognitive Services resource.

Foundry projects underneath an account and individual model deployments do not need the random suffix — they only need to be unique within the parent account:

```
<program>-<workload>-<environment>          # project, e.g. hrl-ai-prod
<model-catalog-name>                        # deployment, e.g. claude-sonnet-5
```

### Log Analytics Workspace

Pattern:

```
log-<program>-<workload>-<environment>-<region>
```

Example:

```
log-hrl-core-prod-wus3
```

### Container App Jobs

Pattern:

```
caj-<program>-<job-slug>-<environment>-<region>
```

Example:

```
caj-hrl-ingest-vector-prod-wus3
```

### PostgreSQL Flexible Server

Pattern:

```
psql-<program>-<workload>-<environment>-<region>
```

Example:

```
psql-hrl-data-prod-wus3
```

PostgreSQL server names must be globally unique.

## Tags

Apply these tags to every resource and resource group:

| Tag           | Value                          | Notes                                    |
|---------------|--------------------------------|------------------------------------------|
| `project`     | `Healthy Rivers and Landscapes`| Always this exact string                 |
| `environment` | `prod`, `dev`, etc.            | Matches the environment variable         |
| `workload`    | e.g. `apps`, `core`, `data`    | Matches the workload local               |
| `owner`       | `HRL Program`                  | Always this exact string                 |
| `managed_by`  | `terraform`                    | Always `terraform`                       |
| `application` | e.g. `restoration-map`         | Add when the resource serves a specific app; omit otherwise |

Tags use human-readable values even when the Terraform local uses an abbreviated form. For example, the bootstrap workload local is `tfstate` but the tag value is `terraform-state`.

## Terraform constructs

### Resource labels

Use `snake_case` for the local label (the second string in a `resource` block). The label should describe the logical role of the resource, not repeat the resource type.

```hcl
resource "azurerm_resource_group" "apps" { ... }
resource "azurerm_static_web_app" "restoration_map" { ... }
resource "azurerm_storage_account" "tfstate" { ... }
```

### Local variables

Use `snake_case`. Define `project`, `workload`, `region_code`, and derived names as locals so names are constructed in one place.

```hcl
locals {
  project     = "hrl"
  workload    = "apps"
  region_code = "wus3"

  resource_group_name = "rg-${local.project}-${local.workload}-${var.environment}-${local.region_code}"
}
```

### Variables

Use `snake_case`. Provide a description for every variable. Use `validation` blocks to constrain allowed values when the set is known.

### Outputs

Use `snake_case`. Match the output name to the Azure attribute it exposes (e.g., `resource_group_name`, `static_web_app_default_hostname`).

## Terraform state files

State keys follow the pattern `<environment>-<workload>.tfstate`:

```
prod-core.tfstate
prod-data.tfstate
prod-pipelines.tfstate
prod-apps.tfstate
prod-ai.tfstate
```

Backend config files in `infra/backend-config/` follow the same pattern with a `.tfbackend` extension:

```
prod-apps.tfbackend
prod-ai.tfbackend
```

## Data storage paths

Within ADLS Gen2 / Blob Storage, organize data using path prefixes. Use lowercase, hyphen-separated names:

```
raw-submissions/
standardized/
  vector/
  tabular/
  raster/
  model-outputs/
validation-reports/
metadata/
catalog/
public-exports/
schema-snapshots/
```
